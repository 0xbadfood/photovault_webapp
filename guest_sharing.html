Guest Sharing Architecture

Invite read-only guests to view your shared photos and albums. Full reference-counted lifecycle with per-host access control.

guest_sharing ‚Äî Design Document v2

üë• Overview

Photovault users can invite guests ‚Äî external users who can view shared photos, videos, and albums in a read-only interface. A guest can be invited by multiple hosts, each managing their own access independently.

Photovault User (Host)

Full Access

  • Upload & manage photos
  • Create albums
  • Share with users & guests
  • Invite / revoke / reactivate guests
  • Face recognition & AI features
Guest (Read-Only)

View Only

  • View shared photos & albums
  • Browse People gallery
  • Browse Discover/Memories
  • ‚ùå Cannot share, edit, or delete
  • ‚ùå Time-limited access

# Reference Counter Flow

The host_count field on the guests table tracks how many hosts have active invitations for this guest. When it reaches 0, the guest account is fully deleted.

1 Host A invites guest@example.com host_count = 1
2 Host B also invites guest@example.com host_count = 2
3 Host A removes guest ‚Äî only A's shared assets cleaned up host_count = 1
4 Host B removes guest ‚Äî account deleted entirely host_count = 0 ‚Üí DELETE

üîí Login Flow

Dual-Auth Login Sequence

1. User submits email + password

2. Server checks user.sql
   ‚Üí Match found? Return { role: 'user', userid: email }

3. No match ‚Üí check guest.sql
   ‚Üí Find guest record by email
   ‚Üí Verify password hash
   ‚Üí Check ANY guest_hosts row with status='active' AND access_till >= now
   ‚Üí Return { role: 'guest', userid: email, hosts: [...] }

4. No match in either DB ‚Üí 401 Unauthorized

üì¶ Asset Tracking ‚Äî guest_assets

When a host shares a photo or album with a guest, a record is created in guest_assets linking the host's asset ID to the guest's copy (symlinked in the guest's per-user DB).

Share Flow

Host shares photo #42 with guest@example.com

1. Existing share_photo() creates symlink + DB record in guest's per-user DB
   ‚Üí guest gets photo_id #7 in their own photovault.db

2. New: INSERT into guest_assets:
   guest_id=3, host_email='host@pv.com', asset_type='photo',
   asset_id=42, guest_asset_id=7

Host removes guest or revokes access

1. Query guest_assets for this guest + host combination
2. For each asset: remove symlinks + DB records from guest's per-user DB
3. Delete guest_assets rows
4. Decrement host_count
‚úì This design cleanly separates per-host shared assets, so removing one host's invitation only cleans up that host's shared content ‚Äî assets from other hosts remain untouched.

# Challenge Solution
