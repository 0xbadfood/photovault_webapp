Daemon Architecture: Two-Thread Split

Splitting the monolithic photo processing pipeline into a fast Scanner thread and a slow AI Worker thread for dramatically improved user experience.

daemonv1.py ‚Äî Design Document

‚ö† Problem

The current daemon.py processes each photo sequentially through the entire pipeline: thumbnail ‚Üí EXIF ‚Üí face recognition ‚Üí description generation. Since face recognition and AI description are very slow (seconds per file), newly uploaded photos cannot even display thumbnails until the AI finishes processing earlier photos in the queue.

‚è± When multiple users upload photos simultaneously, wait times can reach minutes before any thumbnails appear, severely degrading the user experience.

‚ú¶ Solution

Split the processing into two independent threads, each with its own responsibility and run frequency:

Thread 1 ‚Äî Scanner

Fast Path

Runs every 15 seconds. Ensures thumbnails appear immediately.

  • Discover new files
  • Insert into database
  • Generate thumbnails
  • Extract EXIF metadata
  • Determine image type
Thread 2 ‚Äî AI Worker

Slow Path

Runs every 30 seconds. Heavy AI processing at its own pace.

  • Face recognition
  • Face crop & save
  • People matching
  • Description generation
  • (MobileNetV2 inference)

üîí Synchronization Strategy

‚Ñπ Key Invariant: Thread 2 must never process a file until Thread 1 has finished generating its thumbnail and extracting EXIF data.

Database Flags as Barrier

The photos table already has boolean flags: processed_for_thumbnails, processed_for_exif, processed_for_faces, and the description column. These flags serve as the synchronization mechanism:

Thread 1 (Scanner) writes:

UPDATE photos SET processed_for_thumbnails = 1 WHERE id = ?
UPDATE photos SET processed_for_exif = 1, type = ?, date_taken = ? WHERE id = ?

Thread 2 (AI Worker) queries:

SELECT id, path, type FROM photos
WHERE processed_for_thumbnails = 1
  AND processed_for_exif = 1
  AND (processed_for_faces = 0 OR description IS NULL)
  AND type != 'screenshot'     -- skip screenshots
  AND type != 'video'          -- skip videos for face/desc

‚ö° Thread Safety

Concern Solution
SQLite concurrent writes WAL mode enabled (PRAGMA journal_mode=WAL)
Shared connection objects Each thread creates its own connection per scan cycle
AI thread reads half-written row Impossible ‚Äî Thread 2 only reads rows where flags = 1
Race on same file Thread 2 queries only fully processed rows

üèÅ Race Condition Handling

Scenario What Happens
AI thread reaches file before thumbnail exists ‚úì Won't happen ‚Äî DB query filters on processed_for_thumbnails = 1
Both threads write to DB concurrently ‚úì Safe ‚Äî Separate connections + SQLite WAL mode
AI thread crashes mid-processing ‚úì Resilient ‚Äî Flags remain 0, file retried next cycle
Scanner thread crashes ‚úì Resilient ‚Äî Thumbnail flag stays 0, AI ignores it
New files arrive during AI processing ‚úì No delay ‚Äî Scanner picks them up independently
